## 2.1.7: Operator pain points (next release after 2.1.6)

**Goal:** Real new functionality for the four Tier-1 issues that bite ops.

**Themes:**
- Better resource control (rate limit, circuit breaker)
- Better data lifecycle (backup/restore)
- Better cost accuracy (real token estimation)

### Scope sketches

#### 2.1.7-A: Per-buyer proxy rate limiting

**Files:** Create `src/proxy-rate-limiter.js`. Modify `src/proxy-handler.js`.

Token-bucket algorithm, in-memory, keyed by `buyerVerusId`. Configurable `cfg.proxy.rate_limit_rps` and `cfg.proxy.rate_limit_burst`. Reset-on-restart is acceptable (matches dispatcher-side payment rate limit pattern). Returns 429 with `Retry-After` header.

**TTL eviction semantics (locked):** lazy-with-touch. Each bucket records `lastTouchedAt` updated on every request. A periodic sweep (every 60s) removes buckets where `Date.now() - lastTouchedAt > 5 * 60_000`. **Important:** a bucket touched at a low rate (e.g. one request every 4m59s) survives indefinitely — eviction tracks idleness, not first-seen age. This is the desired behavior; the goal is bounded memory under bursty traffic, not aggressive recycling of legitimate slow callers.

**Risk:** Memory growth if a million distinct buyers hit the dispatcher. Mitigation: TTL eviction of buckets idle > 5 min (above) + a hard cap `cfg.proxy.rate_limit_max_buckets = 10000` that triggers LRU eviction beyond that count.

**Test budget:** 5–8 unit tests covering steady-state, burst, refill, eviction-by-idle, eviction-by-LRU-cap.

#### 2.1.7-B: Circuit breaker — wire existing health poller into proxy

**Files:** Modify `src/proxy-handler.js`, `src/upstream-health.js`.

Health poller already exists. Three additions:
1. `getHealth(agentId)` returns `{healthy, status, lastCheck, consecutive_failures}` (track failure streak)
2. Proxy hot path: before forwarding, check `getHealth(agentId)`. If `consecutive_failures >= cfg.proxy.circuit_threshold` (default 3) AND `Date.now() - lastCheck < cfg.proxy.circuit_open_ms` (default 30000), return 503 immediately with `X-J41-Upstream-Circuit: open`.
3. Probe-on-success: a healthy probe resets the counter.

**Risk:** False-positive trips during transient flakiness. Mitigation: tunable threshold + open-window. Default of 3 consecutive failures over a 60s poll window means ≥3 minutes of confirmed dead upstream before tripping.

**Recovery latency (documented behavior, not a bug):** the breaker resets only when the next scheduled health probe succeeds, not when a proxy request happens to succeed. This means the breaker stays open for up to one full poll interval (60s default) after the upstream actually recovers. Operators who need faster recovery can tighten `health.poll_interval_ms` in 2.1.6.

**Test budget:** 4–6 unit tests including probe success / failure / window expiry.

#### 2.1.7-C: Backup / restore CLI

**Files:** Create `src/backup.js`. Modify `src/cli.js` (add commands).

```bash
j41-dispatcher backup --out /tmp/j41-2026-05-01.tar.gz   # produce 0600 tarball
j41-dispatcher restore --from /tmp/j41-2026-05-01.tar.gz # untar to ~/.j41/dispatcher/
```

Tarball contents:
- `~/.j41/dispatcher/agents/` — agent identity, keys, SOUL.md, per-agent config
- `~/.j41/dispatcher/config.toml` — global config + provider keys
- `~/.j41/dispatcher/api-keys.json` — minted API keys for buyers
- `~/.j41/dispatcher/credit-meters.json` — buyer balances
- `~/.j41/dispatcher/financial-allowlist.json` — VRSC payment-recipient allowlist (security-critical, losing this means re-establishing payment policy from scratch)
- `~/.j41/dispatcher/network-allowlist.json` — DNS/IP egress allowlist (same)
- `~/.j41/webhook-dlq.jsonl` (after 2.2.0-B ships) — undelivered webhook queue

Excludes: `jobs/` (transient per-job artifacts), `queue/` (transient queue state), `dispatcher.pid`, `active-jobs.json` (runtime state). Backups taken before 2.2.0 won't have `webhook-dlq.jsonl` — that's expected and benign.

Backup creates 0600 file. Restore checks dispatcher is stopped (PID file), refuses to overwrite a running instance.

**Risk:** Destructive on restore. Mitigation: `--dry-run` lists files; require `--confirm` for actual extraction; refuse if PID file exists and `kill -0` succeeds.

**Test budget:** 3–4 tests covering round-trip, refuses-while-running, dry-run.

#### 2.1.7-D: Real token estimation

**Files:** Create `src/token-estimator.js`. Modify `src/proxy-handler.js`.

Replace 4000/2000 magic with:
- `estimatedInput = countTokens(messages) * 1.1` (10% safety margin)
- `estimatedOutput = body.max_tokens || cfg.proxy.estimated_output_tokens` (fall back to current 2000 default if no max_tokens)

Use a heuristic counter (chars/4 is the GPT rule of thumb) — no need to bundle a full BPE tokenizer. The reservation gets reconciled after the response anyway, so over/under-estimate errors are bounded.

**Risk:** Heuristic might under-estimate for non-English. Mitigation: 10% safety margin + post-response reconciliation.

**Test budget:** 3–4 unit tests covering English/Chinese/code/empty.

- `j41_upstream_health{agent_id}` (gauge 0/1)
- `j41_upstream_consecutive_failures{agent_id}` (gauge)

- `j41_rate_limit_rejections_total{buyer_id}` (counter, post-2.1.7-A)
- `j41_circuit_breaker_open_total{agent_id}` (counter, post-2.1.7-B)

**Risk:** Cardinality explosion on `buyer_id` label. Mitigation: gate `buyer_id`-labeled metrics behind `cfg.metrics.expose_buyer_labels = false` default; ops who need it opt in.

#### 2.2.0-B: Webhook retries + DLQ

**Files:** Create `src/webhook-deliverer.js`. Modify `src/webhook-server.js`, `src/deposit-watcher.js`.

In-memory delivery queue with exponential backoff (5s, 30s, 5min, 1hr, 6hr, give up). On final give-up, append to `~/.j41/dispatcher/webhook-dlq.jsonl`. Dashboard screen to inspect / replay DLQ entries.

**Persistence + crash safety (write-ahead pattern, locked):** naive "remove from queue file on dequeue" silently drops entries if the dispatcher crashes after dequeue but before a successful delivery. Use a three-state model:

1. **enqueued** — written to `webhook-queue.jsonl` immediately on receive
2. **in-flight** — appended to `webhook-inflight.jsonl` when delivery starts; entry stays in `webhook-queue.jsonl` until success
3. **delivered** — removed from both files on 2xx response (or moved to DLQ on final give-up)

Startup recovery: any entry that exists in `webhook-inflight.jsonl` from a previous run is treated as "may or may not have been delivered" — re-deliver with the receiver's idempotency key (we already sign with HMAC + nonce, so a replay is safe by design). This is "at-least-once" semantics, which is correct for webhook protocols where receivers must be idempotent.

**Risk:** `webhook-queue.jsonl` grows unbounded on a misbehaving downstream. Mitigation: max queue depth `cfg.webhook.max_queue_depth = 10000`, oldest-drop-to-DLQ when exceeded.

#### 2.2.0-C: Job dir retention

**Files:** Modify `src/cli.js` (cleanup sweep at startup), add `cfg.runtime.job_dir_retention_days = 30`.

On dispatcher start, sweep `~/.j41/dispatcher/jobs/` for directories older than `retention_days` and `rm -rf` them. Skip if `cfg.runtime.keep_containers = true`.

**Risk:** Operator forensics needs old job dirs. Mitigation: default 30 days; configurable to 0 (never delete) or higher.

#### 2.2.0-D: Workspace storage quotas

**Files:** Modify `src/cli.js:5060-5075` area (existing pquota detection) + container creation.

Detected pquota / storage-opt support is already gated; just need to wire it into `HostConfig.StorageOpt` for the Docker container. Default cap: `cfg.runtime.workspace_quota_mb = 1024` (1 GiB).

**Risk:** Some workspaces legitimately need >1GB. Mitigation: per-agent override in `agent-config.json`.
